## ポイント
- ロジックをhook、見た目をUserListに寄せているため、バグが発生した際にどのソースを確認する必要があるかわかりやすい。
- ロジックを分離することで、コンポーネントの記載量が減る
- APIでの受け取りの際は、受信用のクラスを作成し、初期値を設定するようにする。
  - そのようにすることで、フロント側でのundfinedの回避を行うことができる。
- zodを利用するメリットは、実行時に型のチェックが行えるため、TypeScriptの守備範囲以外をチェックすることが可能
  - undefinedのチェックで困るのは基本的にAPIのレスポンス絡み？であれば、APIレスポンス受信の際にzodで型チェック、変換をするのはメリットになりそう。
  - 逆にフロント内部のデータ定義はTypeScriptによって型チェックできているはずのため、zodでチェックするには過剰になる可能性が高い。
- そもそも全く想定外のAPIレスポンスが帰ってきた場合(safeParseでエラー)はエラーハンドリングを行う。

## Reactの基本的な考え方
- State management with hooks がベースとなる。
  - Hookにビジネスロジックを寄せ、コンポーネントはあくまで画面表示を担う。
